# 被告警吵醒太多次，我做了个让告警自动修复的监控工具

上个月某个深夜，我的服务器挂了。不是第一次，也不会是最后一次。告警发来，ssh 上去，`top` 看一眼，翻日志，猜原因——整个流程做了太多次，终于决定让这件事自动化。

做了个开源工具 VigilOps，核心想法是：告警触发后，让 AI 先分析原因，能自动修的就自动修。分享一下，欢迎提意见。

GitHub: https://github.com/LinChuang2008/vigilops

---

## 为什么做这个

我知道 Grafana、Prometheus、Zabbix 都挺好的，其实 Grafana 的图表做得比我好看多了，Prometheus 的生态也更成熟。

我的问题不是数据不够，是**每次出问题我都要自己去猜原因**。

告警响了，CPU 95%，然后呢？ssh 上去，`top` 看一眼，日志翻一翻，如果是见过的问题还好，如果是第一次遇到的，二十分钟过去了还在猜。这个过程我做了太多次，开始想能不能让机器先帮我走一遍。

另外就是告警疲劳。配置得越细，告警越多，看多了就不想看了。这个问题我自己踩过——某次线上出问题，日志里早有异常，但被淹在一堆告警里没注意到。

所以我想做的不是又一个看板工具，而是**从告警到行动这段路，能不能短一点**。

---

## 主要功能

### AI 根因分析

告警触发后，系统会把当前指标、近期日志、历史模式打包给 AI，返回一个分析结果。

不是那种没用的"请检查系统资源"，而是尽量给出具体的判断和建议，比如：

> "过去 2 小时内存持续线性上涨，无明显峰值回落，符合内存泄漏特征。上周同样模式出现过，当时重启 nginx 后恢复。建议先执行 `nginx -s reload`，同时排查第三方模块。"

AI 用的是 DeepSeek API（也支持 OpenAI），分析不保证 100% 准，但至少给一个方向，比从零开始猜要快。

### 自动修复 Runbook

有些处理方式是固定的：nginx 挂了重启、磁盘快满了清日志、服务无响应重启并通知。

内置了 6 个常见 Runbook，也可以自定义。可以配成**全自动**，也可以配成**AI 分析 + 人工确认**——后者是我自己用的模式，它先分析和准备好操作，我确认一下再执行，不完全放手但也不用自己翻日志。

这两种模式各有适合的场景，生产环境我更建议先用确认模式。

### MCP Server

这个可能是比较少见的功能。VigilOps 提供了一个 MCP Server，可以接入支持 MCP 协议的 AI 工具，比如 Claude Desktop。

配好之后可以直接问 AI：

> "我的服务器现在怎么样？"  
> "过去一小时有异常吗？"

不用打开看板，不用 ssh，适合在手机上快速确认状态。这个功能相对小众，但我个人觉得挺实用的。

### 部署

```bash
git clone https://github.com/LinChuang2008/vigilops.git
cd vigilops
cp .env.example .env
# 填入 AI API Key 和数据库密码
docker compose up -d
```

Docker Compose 部署，正常十几分钟能跑起来。

## 一个具体的例子

模拟了一个 nginx 内存泄漏的场景：内存使用率持续攀升，触发 80% 阈值。

告警后 AI 给出的分析大概是：线性增长曲线符合内存泄漏特征、排除了流量突增的可能、和历史某次异常模式相似，建议 `nginx -s reload` 先释放内存，然后查模块配置。

点确认，nginx 重启，内存回正常。**从告警触发到处理完成，差不多 2 分钟，全程没开终端。**

当然，这是测试环境，问题也比较典型。真实生产环境的问题往往更复杂，AI 分析可能不准，这个要说清楚。但对简单和中等复杂度的问题，这套流程确实有用。

## 现状，说实话

项目还比较早期，不少地方还在完善：

- 采集 Agent 的配置还有些繁琐，第一次上手可能要摸索一会儿
- AI 分析效果和你用的模型、喂的数据质量有关，不是银弹
- 文档还在补，部分高级功能说明不够详细
- 没有在大规模生产环境验证过，只在小规模使用

我自己在用，但我的场景是几台服务器的个人项目，如果你是管几百台的运维，可能还不够稳。
