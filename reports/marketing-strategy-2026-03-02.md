## 核心判断：先找到"那100个人"

不要想着让所有 DevOps 工程师知道 VigilOps。  
先找到 **100 个愿意在生产环境部署并给出反馈的人**。  
他们是你的传播起点，不是你的终点。

---

## 一、破局点分析

### 最小可行受众画像（精准定位）

**不是所有运维工程师**，而是：

> 一个人负责 3-10 台服务器、每周被告警叫醒 2 次以上、用过 Prometheus/Grafana 但觉得"太重了"的独立开发者或小团队 DevOps。

他的痛点不是"监控功能不够"，而是 **"告警太多，修复靠手动，半夜被叫起来"**。

VigilOps 解决的正是这个问题。这就是破局点。

### 为什么不是中大型企业？

- 中大型企业有专职运维团队，有 Datadog 预算，决策链长，不是早期采用者
- 独立开发者 / 小团队：决策快、愿意试新工具、会在社区分享

### 先中国还是国际？

**先中国，30天内。国际市场 60 天后再推。**

理由：
- 中文内容你写得快，回复社区问题成本低
- MCP 概念在国内 AI 社区（Cursor 用户群、Claude 用户）已经有认知基础
- 掘金/V2EX/微信群能快速验证 PMF（产品市场匹配）

## 四、MCP 差异化话语体系

### 核心问题：大多数人不懂 MCP

**绝对不能说：**
> "VigilOps 是首个原生支持 MCP 协议的开源监控平台"  
（技术词汇堆砌，用户不知道为什么应该在乎）

**应该说：**

### 一句话差异化（对普通运维工程师）
> **"Grafana 告诉你哪里坏了，VigilOps 帮你修好它。"**

### 一句话差异化（对 AI 工具用户 / MCP 已知人群）
> **"把你的服务器接入 Claude/Cursor，用自然语言问它'现在有什么问题'，然后让它直接修。"**

### 向完全不懂 MCP 的人解释（30秒版）
> "你用过 ChatGPT 吗？现在有一个标准叫 MCP，让 AI 助手可以直接连接外部工具。  
> VigilOps 实现了这个标准——意味着你可以在 Claude 或 Cursor 里直接问'我的服务器现在怎么样'，AI 会实时查询你的监控数据，然后帮你执行修复命令。  
> 不是聊天，是真的操作服务器。"

## 六、成功指标

### 不要看的指标（虚荣指标）
- ❌ 文章阅读量
- ❌ GitHub Stars（除非对应真实部署）
- ❌ 网站 UV

### 要看的指标（真实指标）

| 指标 | 30天目标 | 测量方式 |
|------|----------|----------|
| **真实部署用户** | ≥ 20 | 用户群 / GitHub Issue 询问 |
| **Demo 活跃用户**（登录 > 5 分钟） | ≥ 50 | 服务器日志 |
| **GitHub Stars** | ≥ 50 | GitHub |
| **收到用户反馈/Issue** | ≥ 10 | GitHub Issues |
| **1v1 用户访谈** | ≥ 3 | 直接联系 |
| **发现 PMF 信号** | 至少 1 个"我们在生产用了" | 用户群/邮件 |

### 关键判断：第 30 天的 PMF 测试

问自己：**"有没有用户说'如果 VigilOps 停止维护，我会很难受'？"**

- 如果有 → 继续往第二阶段走，找更多这样的人
- 如果没有 → 回到用户访谈，搞清楚痛点没找对还是产品没解决
